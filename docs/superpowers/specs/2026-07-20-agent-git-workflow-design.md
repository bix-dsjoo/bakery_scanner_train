# 에이전트 전용 Git 작업 흐름 설계

## 목적

이 저장소는 모든 코드 변경, 검증, 코드 리뷰, PR 생성과 `main` 병합을 AI 에이전트가 수행한다. 사람의 수동 승인을 전제로 하지 않고도 변경의 출처, 검증 근거, 리뷰 독립성과 되돌리기 경로를 확보하는 Git 작업 흐름을 정의한다.

이 설계는 데이터와 평가 정책을 변경하지 않는다. 기존 `AGENTS.md`의 데이터 격리, 모델 경계, 평가 및 CPU benchmark 규칙은 모든 Git 작업 규칙보다 우선한다.

## 선택한 전략

짧은 작업 브랜치, PR, 독립 에이전트 리뷰, squash merge로 구성된 GitHub Flow를 사용한다.

- `main`은 유일한 장기 브랜치이며 항상 병합 가능한 상태를 유지한다.
- 모든 변경은 최신 `main`에서 만든 작업 브랜치에서 시작한다.
- 작업 브랜치 하나와 PR 하나는 하나의 검증 가능한 목표만 다룬다.
- 작업 브랜치의 작성 에이전트와 리뷰 에이전트는 서로 다른 에이전트 인스턴스여야 한다.
- `main`에는 squash merge로 하나의 의미 단위만 기록한다.

`develop`, 상시 `release`, 상시 `hotfix` 브랜치는 사용하지 않는다. 태그 또는 별도 단기 브랜치는 특정 배포·복구 작업에만 사용하며, 기본 개발 경로를 늘리지 않는다.

## 브랜치와 소유권

작업 브랜치는 `codex/<type>-<short-description>` 형식을 사용한다. `<type>`에는 `feat`, `fix`, `refactor`, `test`, `docs`, `data`, `experiment`, `perf`, `build`, `ci`, `chore`를 사용한다. 브랜치 이름은 Git 식별자이므로 ASCII 표기를 유지하며, 한국어 문서·커밋 규칙의 예외다.

브랜치를 만든 구현 에이전트만 그 브랜치에 파일을 수정하거나 push한다. 리뷰 에이전트는 새 컨텍스트에서 diff, 테스트 결과와 정책 준수만 확인하며, 병합 전까지 해당 브랜치에 쓰지 않는다. 같은 브랜치를 여러 구현 에이전트가 병렬로 수정해서는 안 된다.

구현 에이전트는 작업 시작 전 최신 `main`을 기준으로 브랜치를 만들고, 병합 직전 다시 최신 `main`을 반영한다. 작업 브랜치를 재작성해야 할 때는 해당 PR의 기존 리뷰와 검증이 무효화됨을 명시한다.

## 커밋과 main 이력

각 커밋은 가능한 한 논리적으로 독립된 변경만 포함한다. 기능 변경과 대규모 리팩터링, 무관한 포맷 변경, 데이터 변경은 같은 커밋이나 PR에 섞지 않는다. 관련 테스트는 해당 동작 변경과 함께 포함한다.

작업 브랜치에서는 중간 커밋을 허용한다. 그러나 `main`에는 squash merge만 허용하며, squash 커밋과 PR 제목·본문은 한국어로 작성한다.

```text
<유형>[선택 범위]: <명령형 한국어 요약>
```

허용 유형은 `기능`, `수정`, `개선`, `시험`, `문서`, `데이터`, `실험`, `성능`, `빌드`, `자동화`, `정리`, `되돌림`이다. 예시는 `수정(데이터): 평가 전용 split 참조를 거부한다`다. 제목은 구현 결과를 구체적으로 설명해야 하며 `wip`, `update`, `fix stuff` 같은 모호한 메시지는 금지한다. 변경 이유, 호환성 영향, 정책 예외는 커밋 본문 또는 PR 본문에 남긴다.

## PR 계약

모든 PR은 다음 정보를 포함한다.

1. 목적: 해결하려는 문제와 변경 이유
2. 변경 범위: 수정한 구성 요소와 의도적으로 제외한 범위
3. 검증: 실행한 명령, 결과, 실행하지 못한 검증과 이유
4. 데이터·평가 정책: test split 격리, 원본 데이터·레지스트리 변경 여부, 파생 데이터 manifest 여부
5. 영향 및 롤백: 호환성, 성능, 데이터, 모델 결과에 대한 영향과 되돌리는 방법
6. 관련 작업: 이슈, 선행 PR, 후속 PR 또는 없음

PR은 작업 중에는 Draft로 둘 수 있다. PR 제목과 본문은 한국어로 작성하며, 리뷰 요청 시점에는 설명을 실제 diff와 일치시키고 모든 필수 검증 증거를 채운다.

## 에이전트 병합 절차

1. 구현 에이전트가 작업 브랜치에서 변경하고, 범위에 맞는 검증을 실행한다.
2. 구현 에이전트가 필수 항목을 채운 Draft PR 또는 Ready PR을 만든다.
3. 독립 리뷰 에이전트가 새 컨텍스트에서 PR 설명, 전체 diff, 테스트 증거와 기존 `AGENTS.md` 정책을 검토한다.
4. 리뷰에서 변경이 필요하면 구현 에이전트가 수정하고, 영향을 받는 검증을 재실행한다. 코드 diff가 바뀌면 독립 리뷰를 다시 수행한다.
5. 병합 에이전트가 최신 `main` 반영 여부, 필수 CI, 해결되지 않은 대화, 최신 검증 증거를 재확인한다.
6. 병합 에이전트가 squash merge를 수행하고, 원격 작업 브랜치를 삭제한다.
7. 병합 실패 또는 배포 후 문제는 새 `fix/` 브랜치에서 되돌리거나 GitHub revert PR을 만든다. `main` 이력 재작성과 force push는 금지한다.

리뷰 에이전트가 이용 불가능하면 병합을 보류한다. 구현 에이전트 자신의 자기 검토는 필수이지만 독립 리뷰를 대체하지 않는다.

## 고위험 변경

다음 변경은 PR에 고위험으로 표시하고, 독립 리뷰에서 관련 정책을 명시적으로 확인한다.

- `datasets/` 원본, COCO JSON, `datasets/class_registry.json`
- `datasets/base/test` 또는 `datasets/incremental/test` 경로를 다루는 코드와 설정
- 학습·평가 split, threshold, checkpoint 선택 로직
- 모델 출력 차원, `model_index` 매핑, detector 고정 원칙
- CI, GitHub workflow, `AGENTS.md`, PR 템플릿, Git 작업 정책

원본 데이터와 레지스트리는 사용자가 명시적으로 요청한 경우에만 변경할 수 있다. 평가 전용 데이터 또는 결과를 학습·모델 선택에 사용하면 안 된다.

## GitHub 적용 체크리스트

문서화한 규칙은 GitHub의 `main` ruleset 또는 branch protection으로 가능한 범위에서 강제한다.

- PR 없이 `main` 갱신 금지
- direct push, force push, branch deletion 제한
- 필수 CI와 상태 검사 통과
- 필수 독립 리뷰 승인
- 새 push 시 기존 승인 무효화 또는 최신 push 승인 요구
- 모든 대화 해결 후 병합
- linear history 및 squash merge만 허용
- 병합 후 작업 브랜치 자동 삭제

GitHub 플랜 또는 권한 때문에 특정 규칙을 강제할 수 없으면, PR 템플릿과 병합 에이전트의 확인 절차로 동일한 요구를 운영상 강제한다. 이 문서는 GitHub 저장소 설정을 직접 변경하지 않는다.

## 산출물

구현 단계는 다음 파일을 추가하거나 갱신한다.

- `AGENTS.md`: 에이전트가 따라야 하는 강제 Git 작업 규칙
- `docs/agent-git-workflow.md`: 절차, 예외, 롤백, GitHub 적용 체크리스트
- `.github/pull_request_template.md`: PR 증거 수집 템플릿

`AGENTS.md`는 간결한 실행 규칙만 담고, 상세 설명은 `docs/agent-git-workflow.md`를 기준 문서로 둔다. 두 문서는 서로 모순되어서는 안 된다.
